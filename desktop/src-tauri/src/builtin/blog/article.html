技术随笔 / No.068 Go技术随笔造轮子 示例内容

用 Go 写了个 300 行的静态博客生成器，然后我把它删了

花了三个周末，从模板引擎写到增量构建，最后发现我真正需要的只是一个把 Markdown 转成 HTML 的函数。这篇聊聊「造轮子」的收益到底在哪里——不在轮子，在造的过程里你终于搞懂了那些一直没看懂的东西。

林知远
2026 年 7 月 14 日 ·约 12 分钟阅读 ·更新于 2026 年 7 月 16 日

2025 年 12 月的三个周末，我用 Go 写完了一个静态博客生成器。300 行，从 Markdown 解析到增量构建，跑一次 0.5 秒。它工作得很好，我用了七个月——然后上个月的一个周二下午，我把整个仓库删了。

这不是一篇「我造了个轮子快来用」的文章。恰恰相反。我想说清楚的是另一件事：造轮子的收益到底结算在哪个账户上——如果你记错了账，同样三个周末，可以是血赚，也可以是血亏。

01一开始只是想少装一个东西

起因很无聊。博客原本用的是一个成熟的生成器，一切正常，直到某天我想改一下标签页的排序方式。翻了四十分钟文档，最后在一个 2019 年的 issue 里找到答案：不支持，而且不打算支持。

那天晚上我算了一笔账。我的需求只有三条：把 content/ 里的 Markdown 转成 HTML、套一个模板、写到 public/。就这三条。为了它们，我的机器上装着一个 60 MB 的二进制，和一整套我这辈子都不会用到的 taxonomy 系统。

工具的复杂度不会消失，它只会从你的代码里搬到你的文档里。

—— 组里的老陈，在一次没人记得起因的争论里

02300 行都花在哪了

第一个能跑的版本只有 96 行

那天晚上 11 点跑通的时候我是真的开心。filepath.WalkDir 走一遍目录，html/template 套模板，os.WriteFile 写出去——标准库把这三件事都做完了，我只是把它们连起来。

真正让它从 96 行涨到 300 行的，是后面那些「顺手加一下」：front matter、标签页、RSS，还有那个后来要了我命的增量构建。删之前最后一版的行数结算长这样：

行数结算.txt 
# gen.go —— 300 行的全部，删掉之前的最后一版
# go run . -in content/ -out public/

阶段        行数    做的事
解析        68 行   走一遍 content/，读 front matter
渲染        94 行   html/template + 一个 Markdown 函数
增量        90 行   比 mtime，决定跳过谁      ← 上个月删的就是这段
写盘        31 行   os.MkdirAll + os.WriteFile
其它        17 行   flag、log、main

# 68 篇文章：全量 0.5s / 增量 0.1s —— 30% 的代码，省下 0.4 秒

你先记住最后那一行注释。整篇文章的关键都在那 0.4 秒里。

03我为一个不存在的问题写了 90 行

90 行，占了整个程序的 30%，用来做一件事：跳过没改过的文章。它做得很漂亮，全量 0.5 秒，增量 0.1 秒。我给自己省下了 0.4 秒。

0.4 秒。写这 90 行花掉我一个周末，按每天构建 20 次算，这笔投资要跑将近 20 年才能回本——前提还是这 20 年里它一个 bug 都不出。

而它出了 bug。

mtime 是个骗子

git clone 之后，所有文件的 mtime 都是 clone 那一刻；git checkout 切分支，会把一批文件的 mtime 改成现在。而我在 CI 里两件事都干了。结果就是缓存永远不命中——增量构建在 CI 上从来没有生效过，一次都没有，而我为它维护了七个月。

我是在给下面这张图收集数据的时候才发现的。

两条线要到几千篇才真的分得开。而我这辈子大概写不到那个数。

04删掉它的那天

上个月的一个周二下午，我在改一个 bug：某篇文章的标签里有个中文逗号，生成出来的 URL 是坏的。改到第二个小时，我停下来问了自己一个问题——我到底在维护什么？

  1. 它没有用户。除了我，全世界没有第二个人跑过这个程序。
  2. 它没有测试。300 行里一行测试都没有。我一直觉得「这么小的东西不用测」。
  3. 它没有文档。七个月之后，我自己都要先读一遍代码才敢动它。

一个没有用户、没有测试、没有文档，只服务于一个人、每天省下 8 秒的程序。它不是基础设施，它是一只宠物。那天下午我把 content/ 拷了出来，删掉了仓库。

05那三个周末，我到底赚到了什么

如果这笔账记在「我有一个生成器」上，那是血亏——七个月之后它就归零了。但它不该记在那里。真正留下来的是这些：

  • 我终于搞懂了 html/template 的自动转义。用了六年，一直只知道「它会转义」。自己写一遍才明白 template.HTML 为什么必须是显式类型转换——那是一道你必须亲手拆掉的保险。
  • 我第一次真正读懂了 mtime。不是文档里那句「最后修改时间」，而是它在 clone、在 checkout、在 Docker 层里到底会变成什么。这个知识后来救过我们线上一次。
  • 我知道了那 60 MB 里装的是什么。我以为是臃肿，其实大部分是我这三个周末踩过的坑——一个一个，被别人先踩完了。

所以轮子该不该造？该造。但请把它记成一次学习的开销，而不是一次基础设施的投资：前者三个周末很值，后者三个周末远远不够。

造完之后，允许自己删掉它。留在你脑子里的那部分不会 rot。有问题欢迎写信给我，我基本每封都回。
